Machine learning projects often live and die within the confines of a Jupyter Notebook. While notebooks are excellent for experimentation, real-world utility requires production engineering: structured architecture, persistent logging, API-based inference, and containerized deployment.

This car price prediction system is a complete demonstration of that evolution — bridging the gap between a regression algorithm and a consumer-ready web service.

Predicted

The Choice of Random Forest

We selected the Random Forest Regressor as our inference engine. Instead of relying on a single decision tree, it builds a massive "forest" of trees during training and outputs the average prediction.

Multiple models were trained and evaluated, with Random Forest selected for its ability to capture complex non-linear relationships in car pricing.

Why Random Forest? It captures complex, non-linear relationships essentially for car pricing (like the disproportionate drop in value during the vehicle's first 2 years) and is highly resistant to overfitting.

Depreciation Curve

Vehicle prices decay non-linearly with age and mileage. Log-transforming the target variable helps stabilize variance across price ranges.

During training
y_log = log(price)
At prediction time
price = exp(prediction)
Part II — System Architecture

Training vs Inference Separation

Training and prediction are fully decoupled for faster serving, simpler deployment, and reproducible behavior.

Training pipeline
data
  ↓
feature engineering
  ↓
model training
  ↓
saved artifact
Serving system
load model
  ↓
encode features
  ↓
run predictions
  ↓
return response

Model Artifacts

The trained Random Forest pipeline — including feature encoders and the model — is saved as a single joblib artifact and loaded once at startup.

# Artifact path
models/car_latest.joblib

# Contains
feature_encoder   →  categorical encoding
scaler            →  numeric normalization
rf_model          →  trained Random Forest

Configuration & Dependency Injection

Environment variables control all path and runtime configuration, keeping dev, test, and prod cleanly separated. The prediction service receives the model as an external dependency.

MODEL_PATH   = "models/car_latest.joblib"
APP_ENV      = "production"
LOG_LEVEL    = "info"

FastAPI Inference Service

Three endpoints expose the model, health status, and metadata.

POST /predict Accepts JSON car data, returns estimate
GET /health System check for model availability
GET /model-info Version + training metrics

Testing Strategy

Multi-layer testing across API, business logic, feature encoding, and model loader — reaching ~89% code coverage.

# Test layers

Logic Tests      →  prediction wrapper
Loader Tests     →  mocked artifact loading
89% coverage ensures reliable predictions across all vehicle segments and edge cases.

Observability via MySQL

A core requirement for production systems is logging. Every prediction made is stored in a MySQL database. This allows us to track "model drift" and audit the system if any anomalous predictions occur. 

# Schema fields:
[request_id, timestamp, car_features, result, info]

Part III — Frontend Application

Frontend Interaction Flow

When a user clicks Estimate Value, a eleven-step chain connects the browser to the trained model.

1  User enters vehicle specs
2  User clicks Estimate Value
3  JavaScript collects input values
4  JavaScript builds JSON payload
5  Browser sends POST /predict
6  FastAPI validates request
7  Feature encoder runs
8  Model predicts
9  API sends response
10  Saves response to database
11 Frontend displays estimate
